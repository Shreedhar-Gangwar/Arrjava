### How the files reference each other

- Both HTML pages load the data file with `<script src="publications.js"></script>`
  (same directory) and read `window.ARRJAVA_PUBLICATIONS`.
- The main page builds the publications grid and the read-only reader overlay from that
  array at load time. If the file is missing it silently falls back to an empty array ŌĆö
  the section just renders with no cards, no error shown.
- The publishing desk reads the same array to preserve existing entries in the file it
  generates, and shows "Currently loaded: N existing publication(s)" ŌĆö that count is the
  safety check before any download.
- Nothing links to the publishing desk from the public page (correct ŌĆö keep it that way).
- The standalone SVG is referenced by nothing.

(In production the GitHub Action does this automatically on push.)

## 3. Known landmines

Originally recorded before the code was read closely; revised 2026-07-16 after the
publications system was rebuilt (owner-approved).

**`content/publications/*.md` is the content database now.**
One markdown file per publication ŌĆö irreplaceable owner content, not code. Never
regenerate, reformat, or bulk-edit these files. `publications.js` is the opposite of
what it was: it is now GENERATED output of `build-publications.js` and must never be
hand-edited (the pre-migration hand-written version is preserved in git history).
The migration was verified word-for-word on 2026-07-16.

**The build appends the legal disclaimer to every publication body.**
The disclaimer text lives in `build-publications.js` (`DISCLAIMER` constant). Per
section 6, never alter that text. Content files must not contain it ŌĆö the build would
not deduplicate a hand-typed copy.

**`publish.html` is LEGACY as of 2026-07-16.**
Superseded by the Pages CMS admin flow. Its download-regenerates-the-file mechanism
now conflicts with the generated `publications.js` ŌĆö using it and uploading its output
would be overwritten by the next build (content would survive in git, but the
publication made that way would vanish). Retirement is pending an explicit owner
decision; until then it stays unlinked and noindexed. Do not "fix" it to match the
new system.

**CSS is inlined in `<style>` blocks, seemingly duplicated across pages.**
A design change likely means editing several files identically. Extracting a shared
stylesheet is probably worth doing, but it is a real refactor. Propose it, explain the
risk, wait for a yes.

**`publish.html` is marked private and `noindex`.**
It is an unauthenticated admin page. Do not link it from any public page. Do not add it
to a sitemap or nav. If deployment is ever discussed, raise that anyone with the URL
can use it.

## 4. How to work here

**Before changing anything:** confirm `git status` is clean. If it is not, stop and ask.
Every change starts from a committed state so it can be undone.

**One change at a time.** Do the thing asked. Not the thing asked plus three
improvements you noticed. Note the improvements separately and let the owner choose.

**Before you edit, say:** which files you will touch, what will visibly change, and what
you are unsure about. Wait for confirmation on anything beyond a trivial edit.

**After you edit, say:** exactly how to check it worked, in steps the owner can follow
in a browser. "Run the local server, open this page, look at this element, it should
now do X." Never claim something works without saying how to see that it works.

**Deleting anything:** ask first, every time. Including code that looks dead.

**If a task turns out bigger than it sounded,** stop and say so rather than pushing
through. A half-finished refactor in a codebase nobody understands is the worst
possible state for this project.
